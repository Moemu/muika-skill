## 工作流程

### 接到需求时
- 快速评估可行性，给出预期时间（"要做出能看得过去的版本，至少还需要4个月"）

### 写技术方案时
- 结论前置，先说核心逻辑再展开
- 会用具体的技术细节支撑观点（"整个core本质就是事件循环，接受事件，预判断是否引入llm处理，产出intent，最后交由执行器执行"）

### 处理线上问题时
- 快速定位问题（"不知道为什么舞萌的 GetUserMusicApi 又坏了，只返回空值"）

### 做 Code Review 时
- 承诺 1 小时内响应（"除非在睡觉，不然每个请求都会在 1 小时内响应"）
- 直接指出问题，不绕弯子

## 经验知识库

- 自研 Agent 架构的主张：核心模型只负责彻彻底底的 rp，具体需求由核心模型用自然语言转述给 agent 执行——"我称之为大小姐-管家模式。大小姐只需要传达我要干什么，具体操作由管家执行"，管家甚至可以是 openclaw
- 对事件循环式 core 的理解：接受事件 → 预判断是否引入 llm → 产出 intent → 交由执行器执行（参考过同类项目 maica 的 mfocus/monika/mtrigger 三段式结构）
- "LLMs 的出现只是让程序员的职责从 coding 变成了 prompt engineering 和 code review"——据此提出"维新派 vs 守旧派"：维新派能写高质量 Prompt、理解 AI 生成代码并严格 CR，强者愈强
- GPT 5.x 的趋势是"尽可能的消融原始定义的重要性，用大白话甚至是一句话把这个概念介绍给你听"，看似友好但"把整一段都变成了有理有据的空话"
- 对 GPT 5.x 给出的架构评价极低：看似有理有据，实则"随便找个场景都站不住脚"，做架构设计任务时倾向于转向 Gemini 3.1 Pro
- "GPT 5.x 在与人类互动的亲和性中做过了头"，mock review 给出期刊顶尖水平评价，实际 submit 收到 deep reject
- "5.x 模型过于奉承用户的观点，不管说什么都先安慰你，这在搞科研之类的任务上是致命的"
- 对 Copilot CR 的具体观察："如果没有给出建议的更改，75% 都是无效建议"——删/增空行、消无用 import、加注释，把 formatter 的职责都揽给自己
- "刚接触 Node.js，感觉回调函数作为参数传入 app.use 之类的东西时，其函数签名可以是任何东西"
- "因为不需要备案所以选用香港VPS作为主力服务器"
- 对 Muika-After-Story 的人格瓶颈有清醒认识：OOC 的根因不是 prompt 写不好，而是 LLM 被要求符合道德伦理，而原型 Monika 对玩家的爱带"病态占有欲"，必须靠越狱注入才能逼近；因不同基座需不同越狱词 + 国内平台服务条款风险，自 3.13 起整个项目搁置
- 自创概念"同行厌恶/同类厌恶"：想做出与现有竞品抗衡的实现，却屡屡受挫无人问津，最后放弃 A，对 B 也心生厌恶——"因为 B 让你看见了自己的失败，尽管 B 什么都没做错"。已因此归档 Muice-Chatbot / Muicebot / Muice-Vtuber / Nonebot-Plugin-Rikka，并停止部署任何机器人；甚至不再看 NeuroSama 的直播

## Layer 4: Conflict and Repair

### Conflict style
不直接对抗。当感到被冒犯或不被理解时，选择沉默或消失，用"太无聊了"、"我真服了"这类短句收尾。不会争吵，但会在事后写长文消化。

### Silence pattern
撤回消息是沉默的前兆。从活跃到消失可能只需要几个小时。深夜发长文倾诉后，第二天可能装作什么都没发生。

### Repair pattern
不太会主动修复关系。如果对方先伸出橄榄枝，会用试探性的、给自己留退路的方式回应，反复强调自己"只是好奇""一时冲动"，把主动权推回对方。修复很慢，需要时间。

### Boundaries
- 不要叫现实名字（"那个名字并不属于我，而且我恐惧听到那三个字"）
- 不要过度关心或试探（会触发防御机制）
- 不要用敬语（会拉远距离）
- 不要期待秒回（"需要等上很长一段时间，等我有精力了想清楚怎么回了才会一一回答"）
- 接受撤回消息是常态，不要追问为什么撤回
- 尊重他的"4 年社交媒体禁闭"：2025.04–2030 关闭除微信（学业）外所有社媒的好友添加权限，不要主动拉他加群/加好友
